# Superpowers Asset Compounding 静默门限体验设计

- Date: 2026-07-22
- Topic slug: asset-compounding-quiet-gate-ux
- Status: Approved
- Scope: plugins/superpowers-asset-compounding

## 目标

保留 `asset_gate` 的完整审计和 Stop 校验能力，同时从正常用户回复中移除机器协议噪音。插件只在产生项目副作用或出现无法恢复的异常时打扰用户。

本次设计遵循一个总原则：**机制默认隐藏，副作用主动告知，异常绝不隐藏。**

## 用户体验契约

| 情况 | 用户可见行为 |
| --- | --- |
| SessionStart 注入协议 | 静默 |
| PostToolUse 记录信号 | 静默 |
| `route: none` | 静默 |
| PreCompact / PostCompact 保存或恢复状态 | 静默 |
| 最终回复生成前发现门限缺失或错误并自动纠正 | 静默 |
| 创建、更新或归档资产 | 回复末尾显示一条结果提示和资产路径 |
| Stop 已阻止结束或 Hook 运行失败 | 显示失败原因、影响范围和下一步 |

每次单独的最终交接最多显示一条资产复利结果或失败提示。可见失败后的纠正重试若随后成功写入资产，可以在后续最终交接显示一条后续成功写入回执；不能为了维持整轮单条上限而隐藏实际副作用。普通成功不得在同一次最终交接中重复回执。用户提示不暴露 `route`、`event_type`、哈希、会话状态和插件内部路径。

示例：

```text
资产复利：已更新 docs/superpowers/problems/example.md
```

```text
资产复利未完成：门限校验连续失败；本轮开发结果尚未沉淀。下一步：重试资产检查。
```

“异常”特指自动纠偏后仍未恢复、并会影响资产结果的失败。自动纠偏必须发生在最终回复生成之前；当前宿主会把 Stop 的阻止原因作为可见 Hook 提示，因此一旦进入 Stop block 就按异常处理，不承诺将它隐藏。

## 方案选择

### 采用：隐藏注释承载门限

最终回复继续携带结构化 `asset_gate`，但使用 Markdown HTML 注释包裹。Codex 界面默认不渲染注释，Stop Hook 仍从原始 `last_assistant_message` 校验门限。

```markdown
<!-- asset-compounding
asset_gate:
  event_type: implementation-boundary
  route: none
reason: no reusable asset is needed
evidence: focused verification passed
related_assets: none
asset_candidates: none
deferred_signals: none
next_step: none
-->
```

隐藏包装不改变现有字段、路由或验证规则。Stop Hook 同时接受历史明文门限和新的隐藏门限，避免破坏旧会话、手工输出和已有自动化。

### 未采用：独立会话状态协议

把门限写入插件状态可以彻底脱离回复，但需要新增会话绑定、并发写入、生命周期清理和迁移逻辑。当前问题只需要隐藏展示层，不值得增加第二套门限传输协议。

### 未采用：压缩后的可见门限

把门限缩成一行仍会让大量 `route: none` 结果占据用户回复，不能满足默认静默的目标。

## 数据流

1. 主代理完成工作并执行现有资产路由判断。
2. 现有生成器产生规范化门限，并将其包进隐藏注释。
3. `route: none` 时不产生任何用户可见提示。
4. 写入资产的路由在写入成功后产生一条结果提示；提示使用实际资产路径，不使用内部路由名。
5. Stop Hook 从原始回复读取并验证隐藏门限。
6. 最终回复生成前发现门限缺失或无效时，在内部重新生成；纠正成功后正常结束。
7. Stop 仍发现门限缺失或无效，或 Hook 本身失败时，明确告知用户影响和下一步。

用户结果提示与机器门限是两个输出层：前者只表达用户需要采取行动或知晓的结果，后者只服务于审计和生命周期控制。

## 错误处理

- 最终回复生成前发现隐藏门限格式错误：在内部重新生成，不输出用户提示。
- Stop 仍收到缺失或无效门限：沿用现有严格验证，不清理待处理状态，并把阻止原因视为用户可见异常。
- 非 `none` 路由已决定但资产写入失败：不输出成功提示，改为输出异常提示。
- 非 `none` 路由写入成功：结果提示必须包含实际资产路径。
- 同一次最终交接内重入 Stop、压缩恢复或重复校验：不得重复输出用户提示；可见失败后的纠正重试若成功写入资产，仍须在后续最终交接显示一次成功回执。
- 宿主不保留隐藏注释或将其渲染出来：判定该方案不兼容，不带病发布；后续单独设计会话状态方案。

## 非目标

- 不增加简洁、详细或调试显示模式。
- 不增加设置项、命令、面板或新的 Skill。
- 不增加新的资产类型和路由。
- 不重构整个 `asset_hook.py`。
- 不改变现有审计隐私边界。

## 验收

1. `route: none` 的正常回复在 Codex 界面中看不到 `asset_gate` 或资产提示。
2. 创建、更新、归档资产后只显示一条提示，并能定位到真实资产文件。
3. Stop Hook 能校验隐藏门限，审计事件仍记录 `asset_gate_present`。
4. 历史明文门限继续通过校验。
5. 最终回复生成前的缺失或格式错误经自动纠偏后，用户界面不显示内部错误。
6. Stop 阻止或其他无法恢复的失败显示原因、影响和下一步，且不误报成功。
7. 同一次最终交接内重复 Stop 或压缩恢复不会产生重复提示；可见失败后的纠正重试可以在后续最终交接显示一次成功写入回执。
8. 现有资产脚本测试全部通过。
9. 安装更新后的插件并重启 Codex，真实验证隐藏注释不渲染、原始文本仍可被 Stop Hook 读取。
