# engineering-workflow 实测报告

- 日期：2026-07-22
- 项目：Measurement Window
- 技术栈：.NET 10、标准库、单一 Console 项目
- 目标：用一个包含解析、窗口边界、数值可靠性和 CLI 错误语义的小项目走完工程闭环

## 实际路由

| 技能 | 实际作用 | 结果 |
|---|---|---|
| `clarify` | 在缺少指定项目时采用可逆默认，明确输入、输出和失败边界。 | 没有用低价值问题阻塞执行。 |
| `spec` | 在对话中形成验收条件，没有为单会话示例另建规格文件。 | 文档量合适，但首次遗漏空窗和部分输出语义。 |
| `plan` | 使用五步执行计划，没有拆 Ticket 或创建 solution。 | 粒度合适。 |
| `build` | 创建单项目、标准库实现和自检。 | 首次构建暴露测试 lambda 编译错误。 |
| `debug` | 用构建命令稳定复现，定位三个错误的共同根因。 | 一次修复全部调用点。 |
| `review` | 按规格、工程质量、可靠性审查。 | 发现有限输入求和仍可能溢出，改为在线均值并补回归。 |
| `verify` | 运行构建、自检、格式检查、CLI 正反向 Smoke Test 和工作区检查。 | 发现根 `.gitignore` 缺少 `bin/obj`。 |
| `explore` | 未触发。 | 没有外部事实或原型决策，跳过是正确行为。 |

## 验收证据

- Release build：成功，0 warning，0 error。
- Self tests：8/8 passed。
- Format：`dotnet format --verify-no-changes` 通过。
- 正向 Smoke Test：三个输入值生成两个 JSONL 窗口，统计值正确。
- 反向 Smoke Test：乱序输入报告 `Line 2` 并返回退出码 1。
- 晚错误语义：退出码 1 前保留已输出的完整窗口，README 已明确要求调用方检查退出码。
- Workspace：`bin/`、`obj/` 已忽略，`git diff --check` 通过。

## 需要优化的点

### P1：缺少技能组级入口

用户说“使用该技能组”时，没有一个技能负责选择和串联阶段；本次由执行者手工编排。README 不会作为技能触发上下文自动加载。

建议增加轻量内部入口 `workflow` 或 `index`：只判断任务规模、选择最小技能链、维护当前阶段，不复制八个技能正文，也不要求用户先学会选命令。

### P1：阶段之间缺少共享任务契约

`spec` 输出没有要求 AC 编号，而 `verify` 模板直接使用 `AC-1`；`plan`、`build` 和 `review` 也没有统一引用方式。本次验收条件只能靠对话上下文传递，并首次漏掉“空窗是否补零”和“晚错误是否允许部分输出”。

建议让 `clarify/spec` 输出同一个紧凑契约：目标、假设、`AC-*`、边界行为、非目标、风险。后续技能只消费和更新这份契约，不新增另一套文档。

### P1：缺少可回归的技能评测

当前只有 Markdown 和结构校验，无法证明模型是否选对技能、是否过度生成规格、是否在 review-only 请求里改代码。

建议加入少量 eval cases：每条包含用户提示、应触发技能、不应触发技能和关键输出约束。优先覆盖小改动、普通功能、疑难 Bug、外部研究、只审查、只验收和“使用整个技能组”。

### P2：`build` 的 RED 证据不够明确

技能要求先留下失败检查，但没有要求实际运行并记录失败。本次测试与实现同时落盘，第一次红灯只是测试代码编译失败，不是业务行为失败。

建议对非平凡逻辑要求一句 RED 证据：失败命令和预期失败原因；若任务太小或无法先红，明确写跳过原因即可，不要强制完整 TDD 仪式。

### P2：隐式调用策略需要用数据收敛

八个技能目前全部允许隐式调用，且 `spec`、`plan`、`build`、`review`、`verify` 名称和触发语义较宽。尚未验证同一提示是否会加载过多技能或发生阶段竞争。

建议先用上述 eval cases 观察命中，再决定哪些技能只由组级入口或用户显式调用；不要凭直觉一次性关闭自动路由。

## 总结

核心流程有效：它确实避免了无用规格和 Ticket，真实触发了诊断、可靠性审查和工作区验收，并发现两个代码问题与一个仓库卫生问题。

下一版最值得做的不是增加更多工程方法，而是补齐“组级路由、共享验收契约、可回归 eval”三件事。

本次直接读取本地技能文件执行，尚未覆盖插件安装后真实的自动触发命中率；该部分必须通过独立 Codex 任务评测，不能由本次结果冒充。
