# 技术探索

先写出本次必须回答的一个问题，再选择研究或原型。两者都只服务于决策，不偷偷演化成生产实现。

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。会影响后续工作的结论必须记录到任务文档。

## 选择路径

### 研究

适用于外部事实：SDK、协议、标准、平台限制、产品能力和时效性较强的信息。

- 优先官方文档、标准、论文、源码和厂商资料。
- 对可能变化的事实进行实时核验，并把引用放在对应结论旁。
- 区分资料明确写出的事实、根据事实得出的判断，以及还没有证据的猜测。
- 记录来源事实、工程推断、限制及其对任务的影响；只有证据较长或可复用时才按连续性契约拆到研究目录。

### 原型

适用于状态机、算法可行性、数据流水线或交互方向难以靠讨论决定的情况。

- 逻辑问题优先做最小可运行终端原型。
- UI 问题并列提供少量可切换方案，围绕同一决策比较。
- 原型必须有输入、观测指标和停止条件。
- 将问题、实验、观察、结论、限制和原型代码的处理方式记录到任务文档；内容较长时才拆独立研究文档。删除生产分支上的一次性代码，或者明确写出保留位置。

## 完成输出

输出明确答案、证据、推荐决策、已知限制和下一步。只更新受结论影响的决定、假设、边界、风险和计划；若证据不足，说明还缺哪次实验或哪份一手资料。
