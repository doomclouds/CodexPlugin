# 制定计划

先理解真实代码路径和验收边界，再决定计划粒度。计划是执行导航，不是文档表演。

## 选择最小够用的层级

### 单会话任务

列出 3～7 个可验证步骤。每步包含目标、主要改动位置和最小验证方式。

### 跨会话或多人任务

拆成可独立验收的垂直切片，标明阻塞关系。只有切片能独立领取、并行或恢复时才创建 Ticket。

### 大型且路线不明

先建立决策地图：列出必须回答的问题，以及每个问题需要的研究、原型或用户决策。不要把未知方案伪装成实现 Ticket。

## 规则

- 先检索现有规格、计划、实现和测试，复用仓库已有结构。
- 每个步骤交付一段端到端价值或明确消除一个风险。
- 宽范围迁移使用“扩展 → 迁移 → 收缩”，让中间状态保持可运行。
- 明确依赖、验证命令和停止条件；不要写“处理剩余问题”之类空步骤。
- 不自动创建 Issue、分支或提交，除非用户或仓库工作流明确要求。
- 支持计划工具时同步真实进度；不要把未验证的步骤标为完成。

输出计划后直接指出第一步和当前阻塞项。用户只要求规划时，不开始修改代码。
