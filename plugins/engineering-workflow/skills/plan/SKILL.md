# 定义并规划任务

先确认做什么才算正确，再决定怎样实现。规格和执行计划只是同一任务文档的两部分，不创建平行文件。

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。跨会话任务的目标、验收、计划、进度和下一步都维护在同一份任务文档中。

## 工作流

1. 读取领域文档、已有任务文档、ADR、研究、实现和测试，先校验已经记录的事实。
2. 需求歧义会改变方案时使用 `clarify`；缺少外部事实或需要实验时使用 `explore`，不要把未知方案写成任务。
3. 补齐目标、决定与假设、可观察的 `AC-*`、边界、技术约束和未解决风险。验收条件描述结果，不把实现方式写成需求。
4. 需要多步实施时，拆成可以分别完成和验证的结果；每一步引用相关 `AC-*`，写明依赖、主要改动位置和验证方式。
5. 写明当前状态、阻塞和一个可直接执行的下一步。暂停或恢复任务时更新这些事实，不创建单独的交接文档。

## 按需停止

- 用户只要求规格或设计时，完成目标、验收、边界、约束和风险后停止，不虚构实施步骤。
- 需求和做法已经明确，只缺执行顺序时，复用现有验收条件，直接补计划。
- 只有多人确实要同时领取工作时，才使用仓库现有的任务系统或独立 Ticket。
- 用户只要求规划时，不修改代码。

## 规则

- `AC-*` 一旦被计划、测试或证据引用就保持稳定；需求变化时先更新任务文档。
- 验收覆盖关键失败、空输入、部分结果和取消等适用边界，并从公开入口验证行为。
- 用户要求 TDD，或任务包含核心规则、算法、状态机、解析、并发和高风险错误处理时，标出关键行为的公开测试入口，并在实现时使用[测试驱动开发](../tdd/SKILL.md)。TDD 是实现方法，不单列为项目阶段或验收条件。
- 计划引用验收条件，不重复解释需求。
- 每一步都要产生一个可以验证的结果，或者消除一个明确风险。
- 大范围迁移先增加新路径，再逐步迁移调用方，最后删除旧路径；中间过程必须保持可运行。
- 明确依赖、验证命令和停止条件；不要写“处理剩余问题”之类空步骤。
- 不自动创建 Issue、分支或提交，除非用户或仓库工作流明确要求。
- 每完成一项工作、遇到阻塞或重新拆分任务，都要更新任务文档的状态、验证结果和下一步；未验证的步骤不能标为完成。

输出后指出任务文档位置、第一步和当前阻塞项。
