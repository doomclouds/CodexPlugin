# 审查变更

先固定审查基线：目标分支或提交、当前 diff、原始规格或需求、仓库规则。没有基线时明确采用的范围。

## 三轴审查

### 规格

- 是否完整满足验收条件和关键边界场景；
- 是否误解领域概念、遗漏失败路径或加入未要求功能；
- 测试是否真正证明用户可观察行为。

### 工程质量

- 是否复用现有能力，是否存在重复、散弹式修改或错误抽象；
- 接口是否隐藏复杂度，修改是否落在正确模块和共享根因；
- 错误处理、资源生命周期、并发和取消是否一致；
- 测试是否绑定私有实现，是否遗漏最小回归保护。

### 安全与可靠性

- 检查输入信任边界、鉴权、敏感数据、注入、竞态、背压、超时、重试、部分失败和恢复路径。
- 对 .NET 代码重点检查 `IDisposable`、原生句柄、`CancellationToken`、Channel 上限、P/Invoke、Pinned Memory、设备断连与重连。
- 只报告有具体失败场景和实际影响的问题，不做泛化安全清单表演。

## 输出规则

按严重度列出 findings，先问题后摘要。每条包含：位置、触发场景、影响和最小修复方向；行号必须尽量精确。

如果没有可操作问题，明确写“未发现问题”，并补充未验证环境或测试盲区。除非用户明确要求修复，否则保持工作区不变。
