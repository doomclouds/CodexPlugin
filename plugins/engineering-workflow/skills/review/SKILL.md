# 审查变更

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。开始前明确审查哪个分支、提交或 diff，并读取相关领域文档、任务文档、ADR 和仓库规则。

## 检查内容

### 是否满足需求

- 是否完整满足每条 `AC-*` 和关键边界场景；
- 是否误解领域概念、遗漏失败路径或加入未要求功能；
- 测试是否真正证明用户可观察行为。
- 代码行为、术语或决定是否已超出任务文档、领域导航、相关 `CONTEXT.md` 或 ADR。

### 代码质量

- 是否复用现有能力，是否出现重复代码，或者同一个规则需要修改很多地方；
- 调用者是否需要了解过多内部细节，修改是否放在所有相关调用都会经过的位置；
- 错误处理、资源生命周期、并发和取消是否一致；
- 测试是否绑定私有实现，预期值是否重复实现算法，是否滥用内部 Mock 或调用次数，以及是否遗漏最小回归保护。
- 声称使用 TDD 时，RED 是否确实由目标行为缺失导致，GREEN 是否运行了同一测试；过程证据不能代替最终验收。

### 安全与稳定性

- 检查外部输入、权限、敏感数据、注入、竞态、队列上限、超时、重试、部分失败和恢复路径。
- 对 .NET 代码重点检查 `IDisposable`、原生句柄、`CancellationToken`、Channel 上限、P/Invoke、Pinned Memory、设备断连与重连。
- 只报告能够说明触发方式和实际影响的问题，不罗列与本次改动无关的安全清单。

## 输出规则

分别判断需求、代码质量和安全稳定性，不能因为实现看起来合理就跳过需求证据。

按严重程度列出问题，先写问题再写摘要。每条包含位置、触发方式、影响、关联的 `AC-*`，以及最小修复方向；行号尽量精确。

如果没有可操作问题，明确写“未发现问题”，并补充未验证环境或测试盲区。除非用户明确要求修复，否则保持工作区不变。

跨会话任务只把阻塞后续工作的问题、文档与代码的不一致、未验证风险写入任务文档；完整审查结果按仓库现有方式保存，不复制第二份任务状态。
