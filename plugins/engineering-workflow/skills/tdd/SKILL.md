# 测试驱动开发

一次只推进一个可观察行为。TDD 是能直接完成实现的方法，不是 `build` 前后的额外阶段，也不代替最终验收。产品 `AC-*` 只描述交付行为；RED、GREEN 和 REFACTOR 属于实现证据。

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。跨会话任务先读取相关领域文档、任务文档、代码和现有测试。

## 开始循环

1. 选择一条 `AC-*` 或一个明确行为，确定调用者实际使用的公开入口。
2. 从需求、已确认示例或其他独立依据得出预期结果，不在测试中照抄实现算法。
3. 先写能证明该行为缺失的最小测试，并运行它。

有效的 RED 必须因为目标行为尚未实现而失败。编译错误、测试配置错误、无关失败或从未执行到断言都不算 RED；先修复测试环境，再开始循环。

## RED → GREEN → REFACTOR

1. **RED**：运行目标测试，记录命令和预期失败原因。
2. **GREEN**：只写让当前测试通过所需的最小实现，再运行同一测试。
3. **REFACTOR**：测试通过后再整理命名、重复和结构；每次整理后重新运行相关测试。
4. 当前行为稳定后再选择下一条行为，不一次写完所有测试或所有实现。

## 测试质量

- 从公开输入和输出验证用户或调用者关心的结果，不测试私有方法。
- 预期值应来自固定示例、明确规则或其他独立来源，不能用与实现相同的计算过程生成。
- 行为在数量、时间、长度或状态阈值处切换时，至少覆盖阈值本身和紧邻两侧；不要只测远离边界的普通样例。
- 优先使用真实的内部协作者；只在外部 API、数据库、文件系统、时间、随机数、设备等系统边界使用替身。
- 除非调用次数或顺序本身就是公开约定，否则不要把它当作主要断言。
- Bug 修复的第一条测试必须能识别用户报告的症状，而不只是覆盖修改后的函数。
- 测试应能承受内部重构；如果行为没变却频繁破坏测试，测试位置可能不对。

## 不适用时

- 文案、机械映射、生成代码、一次性原型和显然的一行修改不强制 TDD。
- 无法建立稳定测试环境时，说明限制，运行最接近真实行为的检查；不要伪造 RED。
- 用户明确要求 TDD 但当前环境做不到时，先报告阻塞和最小解法，不悄悄改成测试后补。

跨会话任务把 RED、GREEN、整理后的验证结果和下一行为写进现有任务文档的“进度与证据”，不创建独立 TDD 文档。

完成时列出每个行为切片的 RED 命令与失败原因、GREEN 结果、相关回归验证以及仍未覆盖的环境。
