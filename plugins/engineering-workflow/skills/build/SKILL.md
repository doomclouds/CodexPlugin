# 实现需求

先顺着真实调用路径看清楚代码，再选择最小且正确的改动位置。

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。跨会话任务先读相关领域文档和任务文档，再用代码、测试与 Git 确认当前状态。

## 工作流

1. 确认要完成什么、哪些情况必须处理、如何验收，以及工作区是否已有未提交改动。没有 `AC-*` 编号时，从用户请求中整理出最少的验收条件。
2. 查找已有帮助函数、模式和依赖，优先复用；不要为未来可能性增加抽象。
3. 把改动放在所有相关调用都会经过的位置。测试尽量通过公开输入和输出验证行为，不绑定内部实现。
4. 一次完成一个可以单独运行和验证的结果，然后运行最相关的测试或编译。
5. Bug 修复要追到共享根因并检查所有调用方，避免给每个症状路径各加补丁。
6. 完成后根据改动风险扩大测试范围，并检查是否误改了无关文件。
7. 跨会话任务每完成一项工作或遇到阻塞，都要更新任务文档中的状态、验证结果和下一步。

## 实现约束

- 简单映射、文案或显然的一行修改不强制 TDD。
- 如果改动包含条件分支、循环、解析、并发、金额、安全，或者错误可能造成数据丢失，至少添加一个可运行的回归测试。
- 写实现前先运行这个测试，并记录 `RED: <命令> → <预期失败原因>`。如果不适合先写失败测试，记录 `RED skipped: <原因>`；测试代码本身报错不算有效失败。
- 外部输入校验、数据保护、安全措施、可访问性和防止数据丢失的错误处理不能省略。
- 工作区已有未提交改动时，不要覆盖或回退它们。
- 如果需求、领域术语或重要决定发生变化，先更新对应文档，再继续修改代码。
- 不自动提交、推送或创建 PR，除非用户或仓库规则明确要求。

## 完成输出

先说明实现结果，再按 `AC-*` 列出实际运行的验证和剩余风险。环境不允许完整验证时，明确区分“未运行”和“已失败”，不要宣称已经完成。
