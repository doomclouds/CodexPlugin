# 实现需求

先读懂端到端调用路径，再选择最小正确改动。短 diff 只有落在正确接缝上才有价值。

## 工作流

1. 确认任务边界、验收条件、仓库规则和工作区现有改动。
2. 查找已有帮助函数、模式和依赖，优先复用；不要为未来可能性增加抽象。
3. 找到公共行为接缝。核心业务、复杂分支、算法和高风险链路先留下一个会失败的检查。
4. 按垂直切片推进：一次完成一段可运行行为，再执行最小相关测试或编译。
5. Bug 修复要追到共享根因并检查所有调用方，避免给每个症状路径各加补丁。
6. 完成后运行与风险相称的扩大验证，并检查是否误改了相邻文件。

## 实现约束

- 简单映射、文案或显然的一行修改不强制 TDD。
- 非平凡分支、循环、解析、并发、金额、安全或数据丢失路径至少留下一个可运行回归检查。
- 不省略信任边界验证、数据保护、安全措施、可访问性和防止数据丢失的错误处理。
- 尊重脏工作区；不覆盖、不回退用户已有改动。
- 不自动提交、推送或创建 PR，除非用户或仓库规则明确要求。

## 完成输出

先说明实现结果，再列出实际运行的验证和剩余风险。若环境阻止完整验证，明确区分“未运行”和“已失败”，不要宣称完成。
