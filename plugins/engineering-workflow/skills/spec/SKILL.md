# 编写规格

综合现有对话和仓库事实，不重新采访用户。只有缺失信息会实质改变规格时才补问。

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。跨会话任务更新同一份任务文档，不另建重复规格。

## 工作流

1. 通过 `docs/domain/CONTEXT.MAP.md` 读取相关领域的 `CONTEXT.md`，再读取任务文档、ADR、研究、现有实现和测试，优先更新已有资产。
2. 区分需求事实、已经确定的方案、技术限制、假设和尚未解决的风险。
3. 用用户或调用者能观察到的结果写验收条件，并分配不会随意变化的 `AC-*` 编号；不要把实现方式写成需求。
4. 只记录已达成的技术决策，不替未来实现者预先设计所有细节。
5. 将目标、决定与假设、验收条件、不处理的范围、可测试入口和风险写入任务文档对应章节。

保持各节短小：

- 验收条件必须可测试或可观察，使用 `AC-*` 编号，并覆盖关键失败路径。
- 明确空输入、错误、部分结果、取消等特殊情况应该怎样处理。
- 写清楚测试应从哪个公开入口验证什么行为，不绑定私有实现。
- “不处理的范围”描述整个功能不会交付的相邻需求；“当前会话只规划、不实现”只是本次会话的限制，不能写成产品范围。
- 风险只记录尚未解决且会影响交付判断的事项。

完成后指出任务文档位置。实现过程中若需求变化，先更新任务文档再改代码。
