# 编写规格

综合现有对话和仓库事实，不重新采访用户。只有缺失信息会实质改变规格时才补问。

## 工作流

1. 查找相关规格、计划、ADR、领域词汇和现有实现，优先更新已有文档。
2. 区分需求事实、已确认决策、工程约束、假设和开放风险。
3. 用可观察行为写验收条件；避免把某个实现方案伪装成需求。
4. 只记录已达成的技术决策，不替未来实现者预先设计所有细节。
5. 只有任务确实需要持久化时才按仓库惯例写文件；否则直接在对话中给出规格。

## 规格结构

```markdown
# 问题

# 期望行为

# 验收条件

# 领域决策

# 技术约束

# 测试接缝

# 非目标

# 开放风险
```

保持各节短小：

- 验收条件必须可测试或可观察，并覆盖关键失败路径。
- 测试接缝描述应验证的公共行为，不绑定私有实现。
- 非目标明确本次不会顺手完成的相邻需求。
- 开放风险只保留尚未消除且会影响交付判断的事项。

完成后指出规格保存位置，以及下一步是否需要制定计划。
