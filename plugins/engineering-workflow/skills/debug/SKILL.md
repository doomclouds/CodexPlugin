# 诊断问题

如果还没有一个命令、测试、回放或固定操作能够重现用户看到的问题，就不能宣称已经找到原因。

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。跨会话排查先读任务文档，再检查当前代码、测试、日志和 Git 状态。

## 诊断闭环

1. 把症状变成可重复的测试、回放、日志查询、基准或固定输入。
2. 确认当前版本确实失败，并记录最小必要输出。
3. 缩小输入、代码路径、时间窗口或环境差异，保留真实症状。
4. 列出几种可能原因，优先检查最容易区分这些原因、成本又最低的证据。
5. 每次只改变一个条件；使用临时日志或对照实验逐个排除。
6. 继续追踪相关共享函数、状态变化和所有调用方，不要停在第一个异常堆栈。
7. 先添加能复现问题的最小回归检查，再在用户已授权修复时修改代码。
8. 用最初的复现方式和更大范围的测试共同验证修复，并删除临时日志。
9. 跨会话任务要记录复现方式、确认的原因、已经排除的方向、实际验证和下一步要观察什么。

## 边界

- 用户只要求诊断时，停在根因、证据和建议修复，不实施改动。
- 无法复现时，报告已排除内容、缺失证据和下一条最有价值的观测，不编造确定结论。
- 性能问题必须有前后可比较的测量；单次体感不是证据。
- 真实硬件、时钟和传感器允许校准差异；不要用理论常数掩盖物理偏差。

## 完成输出

给出：复现方式、根因、排除过的主要假设、回归检查、修复验证和仍未覆盖的环境风险。
