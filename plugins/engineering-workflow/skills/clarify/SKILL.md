# 需求澄清

只问会改变实现、验收方式或风险的问题。涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。

## 工作流

1. 先读取仓库规则、现有术语、任务文档、相关实现和历史决定，避免询问本可自行发现的信息。
2. 区分三类信息：已经确认、可以暂时假设、必须由用户决定。
3. 每轮最多提出 3 个关键问题，并说明不同答案会带来什么变化。
4. 不影响当前工作的细节可以先写明假设后继续，不要因为小问题停工。
5. 用具体业务场景检查术语含义。一个词表示多个概念时要拆开；术语确认后立即更新对应领域的 `CONTEXT.md` 和导航关系。
6. 最终整理出目标、可观察的验收条件、不处理的范围和剩余风险。

## 文档边界

- 会被后续任务继续使用的领域术语必须写入 `docs/domain/<domain>/CONTEXT.md`，并在 `docs/domain/CONTEXT.MAP.md` 登记或修正导航；没有可记录内容时不要创建空文件。
- 只有影响大、以后很难更改，而且确实比较过不同方案的决定才写 ADR。
- 小改动、临时选择和实现细节不要生成领域文档或 ADR。
- 需要持续推进的任务直接更新同一份任务文档中的目标、决定、假设、验收、边界和风险，不另写澄清摘要。

## 完成输出

简要给出以下结论；没有内容的项可以省略：

- 目标与已确认边界；
- 当前采用的假设；
- `AC-*` 可验收行为；
- 异常、取消等特殊情况；
- 非目标与未决风险；
- 下一步应直接实现、编写规格，还是先做技术探索。
