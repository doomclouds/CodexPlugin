# 需求澄清

把注意力放在会改变实现、测试或风险边界的问题上。不要为了完整感追问无关细节。

## 工作流

1. 先读取仓库规则、现有术语、相关实现和历史决策，避免询问本可自行发现的信息。
2. 区分三类信息：已确认、可采用可逆假设、必须由用户决定。
3. 每轮最多提出 3 个高价值问题，并说明不同答案会改变什么。
4. 对非阻塞信息明确写出假设后继续；不要把轻微不确定性变成停工理由。
5. 用具体场景检验领域词语，拆开含义不同却共用同一名称的概念。
6. 收敛到可观察的期望行为、边界情况、非目标和剩余风险。

## 文档边界

- 只有稳定且会被后续会话复用的领域词汇才写入现有 `CONTEXT.md` 或仓库等价文件。
- 只有重大、难逆并存在真实权衡的决策才写 ADR；先遵循仓库已有位置和格式。
- 小改动、临时选择和实现细节不要生成领域文档或 ADR。
- 未经请求，不要为了这项技能创建新的文档体系。

## 完成输出

简要给出：

- 已确认的任务边界；
- 采用的假设及其可逆性；
- 可验收行为；
- 非目标与未决风险；
- 下一步应直接实现、编写规格，还是先做技术探索。
