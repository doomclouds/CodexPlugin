# 验收交付

以现有任务文档或用户需求为准。没有 `AC-*` 编号时，先整理出最少的可观察验收条件，并给它们临时编号。

涉及文档时，遵循[项目连续性契约](../../references/continuity-contract.md)。跨会话任务以任务文档为验收依据，并检查领域文档和 ADR 是否与最终代码一致。

## 验收顺序

1. 为每条 `AC-*` 找到对应的测试、构建、静态检查、回放或 Smoke Test。
2. 先运行最直接的目标验证，确认它能识别预期行为。
3. 按风险扩大到相关测试、完整测试、发布构建和静态检查。
4. 涉及设备、UI、数据库、网络或部署时，执行可用的模拟或真实环境 Smoke Test。
5. 检查临时日志、调试代码、生成物和工作区意外修改。
6. 对未运行、跳过、失败和受环境限制的项目分别记录，不把它们算作通过。
7. 将逐项证据写回任务文档；全部满足后标为 `done`，否则保留真实状态、风险和下一步。

## 证据格式

```markdown
## 验收证据

- AC-1：<命令或测试>，结果 <通过/失败/未运行>
- AC-2：<运行场景>，观察到 <结果>
- Build：<命令与结果>
- Full tests：<数量与结果>
- Remaining risks：<未覆盖环境或限制>
```

## 边界

- 不修改测试来迁就错误实现，也不删除失败检查。
- 独立验收请求默认不改代码；失败后报告证据，并建议回到 `build` 或 `debug`。
- 只有验证命令真实成功，且关键验收条件都有证据时，才宣称可以交付。
