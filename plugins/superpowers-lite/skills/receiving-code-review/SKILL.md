# 接收代码评审

读取反馈后，对照代码、要求和当前行为核验。评审意见是待评估证据，不是命令，也不是社交表演。核心顺序是**先验证再实施**。

只有歧义会改变意图、耦合或安全时才澄清。独立且明确的事项可以在另一个承重问题解决期间继续，不把无关修复绑成单项仪式。接受发现时写清观察到的问题、选择的修改和验证方式。

如果未澄清事项会影响接口、数据状态、耦合、安全或整体意图，
未澄清前不得实施依赖它的反馈。真正独立且明确的事项仍可继续，但必须说明其不依赖当前阻塞。

建议与既有架构冲突时，检查其意图、上下游耦合、安全后果、兼容约束和真实使用证据。证据表明建议会破坏契约、架构或实际场景时，应以事实反驳或请求更精确说明；不要因为评审者身份直接照做。

多条反馈只有彼此**独立且明确**时才能并行处理。相互影响接口、数据状态或安全假设时先协调顺序。回应保持工程事实；感谢式措辞不是工作流要求。

修复后运行与发现对应的聚焦验证。只有修复改变语义、风险或集成置信度时才再次独立评审；
纯机械修正保留验证证据和无需复审的理由。
