# 请求代码评审

## 目的

独立评审提供实现报告无法提供的视角。把它放在会改变语义行为、集成或风险置信度的位置；不要为了找样式问题或证明流程存在而制造评审。低风险机械工作默认不一定需要独立评审。

## 何时值得评审

这些边界通常值得独立评审：用户可见行为、API 或兼容性、跨模块集成，以及实质性实现里程碑。实质性整体修改收尾前，应让评审者检查组合后的语义行为和集成，而不是只看孤立差异。

短修改涉及安全、数据完整性、权限、并发或跨模块行为时仍需评审。遵循已评审模式的重复低风险机械工作可以分组评审，并保留相称的验收证据。不要机械地把所有改动升级成独立评审。

## 输入与范围

提供允许真实判断的最小持久包：

- 要求或任务简报以及约束；
- 作为待核对证据的执行报告；
- 精确提交范围或工作区的评审包；
- 语义场景、验收证据、集成边界和已命名风险。

要求评审者判断语义行为和集成：是否解决真实问题、保持契约并与相邻模块协作。样式和可维护性只有在威胁当前行为或未来安全修改时才构成发现，不设问题配额。

## 处理发现

依赖评审结果前修复 Critical 和 Important。Minor 只有明确影响、负责人或未来决定时才记录。评审结论与架构冲突时，用代码、要求和运行证据核对，不盲从也不压制。

修复只有在改变语义、风险或集成置信度时才再次评审；局部机械修正保留聚焦验证和无需复审的理由。

使用 Codex 可用的 `spawn_agent` 创建只读评审代理，能力与差异风险相称；禁止固定评审扇出。整体评审模板见 [code-reviewer.md](code-reviewer.md)。
