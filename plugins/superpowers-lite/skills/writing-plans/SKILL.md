# 编写持久计划

## 目的

计划是跨多轮工作、上下文压缩和执行者交接所依赖的**持久执行契约**。假设后续执行者能力足够，但完全没有当前对话上下文；计划必须保存仅凭任务标题无法安全重建的信息：

- 用户意图和真实产品、业务或运维行为；
- 架构、模块关系、数据流和控制流；
- 精确约束、接口、取值、兼容边界与精确文件路径；
- 非例行技术细节、依赖和构造顺序；
- 语义验收场景、证据和回读方式；
- 分支、worktree、提交和推送偏好；
- 中断后恢复所需状态。

计划不是泛泛原则，也不是把每次工具调用抄成流水账。只保留会影响构造、兼容性、验证或恢复的内容。

## 固化事实

1. 读取用户请求、稳定规格、仓库指令、相关实现和测试，以及当前仓库状态。
2. 稳定要求优先引用来源；只在执行点重复不能推断错的精确值、接口、兼容规则和代码细节。
3. 说明行为由哪个模块拥有、边界如何跨越，以及为什么依赖关系决定当前构造顺序。
4. 检查提交、推送、分支和 worktree 偏好。缺少且会改变执行方式时只问一次；没有明确权限不得推送。

否定边界不等于肯定设计。例如“不允许未经批准的远程存储”并未批准本地文件、内存存储或注入式存储。存储、失败策略、目录布局、所有权等选择缺少证据时，必须记录为**开放决策或阻塞**，写明解除条件，并让依赖工作保持条件化。

除非仓库或用户指定其他持久位置，把计划保存为 `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`。

## 划分交付

按可独立验收的有意义交付划分任务。设置、实现、测试、文档和评审可以属于同一交付；只有后续任务依赖一个独立已验收行为时才拆分，不按固定分钟数切碎。

当自动化测试能区分目标行为时使用红绿循环，但编译、lint 和单元测试只是支持证据。每个任务都要从真实场景推导语义验收：

- 前端：检查渲染结果和关键交互；
- 持久化：保存、重启或重新加载后回读；
- API、CLI、自动化：使用有意义参数，覆盖失败路径并检查最终状态；
- 迁移与兼容：用代表性的既有消费者或工件验证承诺边界。

例行代码不写进计划；错误推断会改变行为、兼容性、构造顺序或验收证据时，才加入签名、模式、命令、代码片段或精确值。

## 文档头

每份持久计划以这些字段开头：

```markdown
# [功能名称] 实施计划

**目标：** [用户可见或运维结果]

**稳定引用：** [规格、决策、指令和仓库状态]

**仓库操作：** [已知提交/推送/分支/worktree 偏好，或仍需用户决定的一项内容]

**恢复状态：** [上下文压缩后先读什么、哪些交付可信、还缺哪些证据]
```

## 任务模板

````markdown
### 任务 N：[有意义的交付]

**项目意义：** 此任务解决的用户或运维问题。

**架构关系：** 涉及的模块、所有权边界、数据流和控制流。

**技术契约：**
- 文件：创建、修改、删除的精确文件路径。
- 接口与取值：签名、模式、名称、配置、兼容条件和错误行为。
- 约束：稳定引用，以及本任务不能被不同推断的细节。

**依赖：** 已验收前置、此任务解锁的内容和构造顺序。

**实现说明：** 非例行决策、代码片段、命令和数据形状。

**语义验收：**
- 场景：真实产品、业务或运维用例。
- 证据：测试、命令、渲染状态、持久化回读或其他工件。
- 通过/失败：观察结果的含义，以及必要的负向或兼容情况。

**恢复检查点：** 新执行者在中断、失败或上下文压缩后继续所需的状态、工件和下一决策。
````

## 自审与交接

提交计划前确认：

1. 每项用户要求和稳定约束都映射到任务或明确的范围外决定。
2. 接口、精确值、兼容边界和依赖顺序前后一致。
3. 每个任务都有真实语义验收，而不是只有编译、lint 或单元测试。
4. 新执行者无需重读原始对话即可从恢复状态继续。
5. 仓库操作偏好已记录，或剩余选择已准备好一次性询问。
6. 否定约束没有被偷换成未经批准的肯定设计。

保存后给出基于复杂度、上下文、隔离、成本和长期连续性的推荐，再请用户在 SDD 与当前会话内执行之间选择。用户决定 SDD 时使用 `subagent-driven-development`；选择当前会话执行时使用 `executing-plans`，不要替用户暗选。
