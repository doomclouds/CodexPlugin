# 根因追踪

最初触发点不清楚时，从深层症状反向追踪：记录已观察到的故障，找出它的直接原因，再沿相关值、状态或调用边界向上追，直到证据定位源头或暴露真正未知。优先使用最强适用证据：聚焦复现、堆栈跟踪、边界日志、二分或回读。

源头已知时，在源头修复。如果紧急情况下必须先遏制，明确标成“遏制措施”，记录它没有解决什么，并保持根因调查开放。边界守卫只有在处理独立故障模式时才有价值；它不能替代关于原始触发点的证据。

示例：空目录一路传入深层 `git init`。沿工作区调用链追踪该值，找到提供它的测试或初始化逻辑，在源头修复，并验证预期的工作区行为。拒绝危险目录的守卫可以在调查源头期间遏制附带损害。
