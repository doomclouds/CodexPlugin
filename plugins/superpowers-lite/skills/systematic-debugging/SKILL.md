# 系统化调试

根因、证据、复现和语义验证能防止“看似合理的补丁”变成反复出现的故障。当根因尚不明确、行为确实不清楚、故障重复发生、组件跨越边界，或风险使猜测不安全时，使用此工作流。已知且路径确定的小修复可以直接进行，但仍需匹配风险的验证。

## 调查视角

以下是可以组合的调查视角，不是必须连续执行的固定阶段：观察或复现有意义的症状；检查近期状态与相关边界；对比正常工作的同类对象；追踪输入或控制流；形成单一假设；运行能区分真假的最小实验。只选择能降低当前不确定性的视角，并保存证据。

存在有效自动化测试缝隙时，使用失败测试。否则使用最强适用的复现、反事实或回读。修复根因后，对受影响行为及相关回归执行语义验证。

## 升级与遏制

每次新修复必须由新证据或可证伪的新假设支持；没有新增信息时不得原样重试或继续叠加
补丁。第二次修复失败、症状跨组件迁移，或共享状态证据增强时，必须停止局部修补，重新
检查根因、信任边界和架构假设。这个复盘点不自动证明架构错误，后续方向仍由证据决定。

如果证据显示存在共享状态、隐藏耦合，或每次尝试修复都会移动症状，应在继续堆叠局部改动前重新审视架构。是否升级为架构性处置由证据驱动，不由固定失败次数直接决定。

根因仍在调查时，可以用明确标注的遏制或缓解措施降低损害。它必须说明自身局限；缓解措施不能冒充根治。
