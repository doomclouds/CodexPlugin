# 技能创作最佳实践

本文件保留上游参考文件名，内容已翻译并适配为 Codex 技能创作指南。使用时以当前 Codex 能力、用户指令和仓库策略为准。

## 核心原则

### 保持简洁

上下文是共享资源。默认 Codex 已具备通用推理和编程能力，只写它不能稳定推断的领域知识、脆弱操作顺序、接口约束和验证边界。

差的写法会重复解释常识、堆积同义示例或把工具全部参数抄入正文。好的写法给出一条可靠主路径，并把详细参数交给 `--help` 或链接参考。

### 设置合适自由度

- 多种方案都正确、选择依赖现场判断：给高自由度原则与决策条件。
- 存在推荐模式但允许少量变化：给伪代码、参数化模板或小脚本。
- 操作脆弱、容易破坏数据或必须保持一致：给低自由度脚本、前置检查和失败出口。

自由度应与错误代价匹配，不按文档作者的控制欲匹配。

### 覆盖目标模型与环境

技能可能由不同能力层级的 Codex 模型读取。至少检查：触发描述是否足够具体、低成本模型能否遵循关键边界、高能力模型是否会把开放建议过度扩展。

不要假设平台、shell、依赖或网络总是存在。工具缺失时写明检测方式和安全回退。

## 技能结构

推荐结构：

```text
skills/<skill-name>/
  SKILL.md
  scripts/      # 需要确定性执行时
  references/   # 重型知识按需加载时
  assets/       # 输出所需素材时
```

不要添加 README、变更日志、安装指南或创作过程记录，除非运行面确实引用它们。

### 命名

- 目录和 `name` 完全一致。
- 只用小写字母、数字和连字符。
- 优先使用动词化、能表达任务的短名称。
- 工具专属技能可以用工具名作为前缀以改善发现。

### 描述

`description` 是主要发现入口，必须：

- 以 `Use when` 开头；
- 描述具体触发、症状和边界；
- 使用第三人称语境；
- 避免概括整套流程，让 Codex 必须读取正文；
- 覆盖用户可能搜索的错误、症状、工具和文件类型关键词。

示例：

```yaml
description: Use when 测试存在竞态、时序依赖或结果不稳定。
```

不要写成“用于异步测试”这类抽象标签，也不要把“先做 A 再做 B”塞进描述。

## 渐进式披露

元数据负责发现，`SKILL.md` 负责核心决策，辅助文件负责重型细节。正文接近数百行时，按实际使用分支拆分，并从 `SKILL.md` 直接链接每份参考及其读取条件。

### 高层指南加参考

```markdown
# PDF 处理

## 快速开始
[最常用路径]

## 高级能力
- 表单填写：需要编辑表单时读取 `references/forms.md`
- 底层格式：需要诊断结构时读取 `references/ooxml.md`
```

### 按领域拆分

领域差异大时，把财务、销售、产品等参考分开；正文只解释如何选。Codex 只加载当前任务需要的领域文件。

### 条件化细节

把高级路径绑定到可观察条件，例如“存在修订标记时读取 redlining.md”。不要用多层链接让 Codex 逐级寻宝。

长参考文件在顶部提供目录，方便快速定位。相同内容只保留一处，避免正文和参考漂移。

## 工作流与反馈循环

复杂任务需要明确阶段、输入、输出和停止条件。线性步骤用编号列表，容易提前停止或选错分支时才用流程图。

一个可靠工作流应说明：

1. 读取哪些原始工件。
2. 哪些决定必须由用户或仓库策略提供。
3. 如何产生中间工件。
4. 如何检查中间工件并反馈修正。
5. 哪条证据支持最终结论。

例如文档编辑不应只写“修改文件”，而应包含读取、编辑、渲染、视觉检查、修复和最终回读。中间产物要可检查，不能把所有逻辑藏在一次不可见操作里。

## 内容规范

### 避免易过期事实

版本、产品能力和当前日期会变化。能链接权威来源时使用链接；必须固定值时说明适用版本和刷新条件。不要把“当前”写成永久事实。

### 术语一致

同一概念始终使用同一名称。工具名、文件名和状态值保持精确；用户可见解释可以中文化，但协议字段和值域不要随意翻译。

### 模板

当输出必须包含固定字段时，提供正向模板，并解释每个字段的语义。不要只写一串“不要遗漏”的否定提醒。

### 示例

一个完整、真实、可适配的示例优于五个空壳示例。示例应展示目标边界、失败行为和验证，不泄漏秘密或生产数据。

### 条件工作流

行为取决于明确条件时，把条件写成可观察谓词：

```text
如果任务会修改外部状态，先检查明确授权；否则保持只读诊断。
```

不要写成无条件规则再叠加含糊例外。

## 评估与迭代

### 先定义评估

写技能前先定义它要改变的可观察结果。选择与失败类型匹配的评估：

- 发现问题：给出触发和不触发请求，检查是否加载正确技能。
- 检索问题：检查能否找到正确参考和字段。
- 技术应用：给新场景和边界输入，检查结果。
- 纪律问题：加入现实压力，观察是否越界。
- 输出结构：验证必需字段、顺序和语义。

### 建立反事实

技能声称改变行为时，使用无指导控制判断结果是否来自默认能力。控制组没有失败就不要强行宣布 RED；改进评估或缩小技能价值声明。

### 从真实使用迭代

记录 Codex 在哪里找不到文件、误解条件、选择错误工具、跳过验证或输出错误形状。对准证据修改最小内容，再复测。不要因一个偶然样本堆叠大量禁止条款。

### 观察导航

检查 Codex 是否：

- 从描述正确触发；
- 在正文找到主路径；
- 只加载必要参考；
- 使用脚本而不是重复生成脆弱代码；
- 在正确位置请求权限或停止。

## 常见反模式

### 平台路径写死

命令和代码示例尽量使用跨平台形式。确需 Windows 或 POSIX 语义时明确标注，并避免把一种 shell 的串联和转义直接移植到另一种。

### 选项过多

默认给推荐路径和选择条件。没有实质差异的十种方案只会稀释判断。

### 说明流程而不解决问题

技能应提供可执行方法、脚本或精确参考，不能把困难工作全部推回给用户。

### 深层引用链

辅助文件从 `SKILL.md` 一层可达。参考再引用参考会导致遗漏和上下文浪费。

### 权限暗示

“执行工作流”不等于授权 push、发布、删除或外部写入。所有外部和破坏性操作仍需当前上下文授权。

## 可执行代码

### 解决重复且脆弱的操作

当同一段代码会反复生成，或正确性依赖严格实现时，提供脚本。脚本应该：

- 参数明确并支持 `--help`；
- 验证输入，错误时非零退出；
- 输出可供人和自动化检查；
- 代码和注释使用英文；
- 用户可见信息按插件语言要求本地化；
- 不嵌入凭据、机器专属绝对路径或生产资源。

### 给出关键常量理由

重试次数、超时和限制不能是魔法数字。用英文代码注释解释为什么该值平衡可靠性与成本，但不要把过时环境事实写成永久规则。

### 提供工具脚本

脚本可用于格式转换、验证、抽取、渲染或生成可复查中间产物。正文写明何时运行、输入输出和失败含义，不必复制完整源码。

### 视觉分析

布局相关任务必须渲染并检查。文本提取不能证明页面视觉正确；应生成 PNG、SVG 或其他可见中间产物，并记录检查点。

### 依赖和运行环境

写明必要运行时、包和外部命令。先检测再使用，不假设 `dot`、浏览器、Python 包或 MCP 服务已安装。缺失依赖时提供安装方向或安全降级，不偷偷联网安装。

### MCP 与 Codex 工具

引用当前技能清单中真实存在的 Codex 工具和参数，不保留其他平台专属调用。工具可能随运行环境不同而缺失，因此描述能力与选择条件，并在需要时给出当前 Codex 工具名。

## 技术要求

### YAML frontmatter

至少包含 `name` 和 `description`。frontmatter 必须是有效 YAML，目录名与 `name` 一致，描述简洁且可发现。

### 上下文预算

频繁加载技能尤其要短。把重型 API、模式、示例和工具说明移到辅助文件；删除重复、叙事性复盘和显而易见内容。

## 发布检查清单

### 核心质量

- [ ] 技能适用于跨任务判断，而不是单仓库机械规则。
- [ ] 名称、目录和 frontmatter 一致。
- [ ] 描述以 `Use when` 开头并覆盖具体触发。
- [ ] 正文给出最小安全工作流和语义验证。
- [ ] 指导形式匹配已观察失败。
- [ ] 引用一层可达且无断链。
- [ ] 没有平台专属工具残留或未经授权操作。

### 代码与脚本

- [ ] 脚本已运行或做最强适用验证。
- [ ] 输入、输出和错误退出可观察。
- [ ] 代码及注释英文，用户文案符合插件语言。
- [ ] 没有凭据、生产数据或机器专属秘密。

### 测试

- [ ] `quick_validate.py` 通过。
- [ ] 契约、链接和目录清单通过。
- [ ] 复杂行为有比例适配的真实场景。
- [ ] 声称改变行为时有反事实证据，或明确记录控制组限制。
- [ ] 提交、push、PR 和发布符合当前仓库策略及明确权限。
