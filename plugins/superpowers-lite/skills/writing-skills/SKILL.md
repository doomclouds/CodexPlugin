# 编写技能

只有当可复用指导明显优于项目指令、自动化约束或一次性记录时，才创建技能。机械路径规则如果只服务当前仓库，应进入仓库指令或校验脚本；别什么都封成技能，技能不是文档垃圾桶。

## 最小资产与发现

先选择**最小有意义资产**：通常是包含 frontmatter 和聚焦正文的 `SKILL.md`。只有脚本、重型参考或输出素材能改善真实使用时才增加文件。

目录名与 `name` 保持一致，只用小写字母、数字和连字符。`description` 必须以 `Use when` 开头，只描述具体触发条件、症状和边界，不把完整流程塞进元数据让 Codex 跳过正文。

采用**渐进式披露**：

1. 元数据帮助 Codex 决定是否加载。
2. `SKILL.md` 保存首次执行必需的决策与最小安全工作流。
3. 重型参考、脚本和资产按需从正文直接链接，避免深层引用链。

## 先匹配失败，再写指导

先判断内容是否值得成为跨仓库技能，再识别基线失败：

| 失败类型 | 合适形式 |
|---|---|
| 压力下明知故犯 | 明确禁止、红旗和真实后果 |
| 输出形状错误 | 正向配方或结构契约 |
| 遗漏固定元素 | 模板中的必填字段 |
| 行为取决于条件 | 由可观察条件触发的分支 |
| 机械且可判定 | 自动化校验，不写判断型技能 |

**指导形式必须匹配失败类型**。禁止列表不能可靠塑造输出结构，软建议也不能约束高风险纪律；选错形式会增加歧义。

## 比例验证

验证强度随新颖度、风险和方差变化：

- 参考技能：检查 frontmatter、链接、示例和部署发现。
- 技术技能：用新场景验证能否正确应用。
- 行为或流程技能：在真实压力场景中验证；当结论可能来自 Codex 默认能力时加入无指导控制。

关键行为需要**反事实证据**：没有技能时应出现目标失败，读取技能后结果才发生可归因变化。如果控制组本来就正确，不要伪造 RED；记录限制并改进场景或说明技能只提供结构价值。详细方法见 [testing-skills-with-subagents.md](testing-skills-with-subagents.md)。

## 编辑与质量

- 用命令式中文写面向 Codex 的指令；代码和代码注释保持英文，用户可见输出中文。
- 只保留非显而易见、可执行且跨场景的知识。
- 例子优先完整且可适配，不堆多语言模板。
- 流程图只表达容易选错的决策或循环；规范见 [graphviz-conventions.dot](graphviz-conventions.dot)。
- 清晰触发和真实后果优于权威、稀缺、强制公告或清单表演，见 [persuasion-principles.md](persuasion-principles.md)。
- 通用创作参考见 [anthropic-best-practices.md](anthropic-best-practices.md)，使用时仍以 Codex 能力和当前仓库策略为准。

## 验证与发布

至少运行技能契约测试、Markdown 链接检查和 `quick_validate.py`；脚本必须实跑或做相称语法检查。手工筛查其他平台专属工具、错误平台语境、未翻译用户文案和过期路径。

提交、push、PR、发布、安装和删除都服从**当前仓库策略**与明确授权。格式通过只证明技能可加载，不自动证明真实行为有效。
